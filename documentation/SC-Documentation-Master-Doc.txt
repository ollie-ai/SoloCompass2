# SC-A01 API — Auth Documentation

## 1. Overview
- **Type:** System / Feature
- **Primary purpose:** Secure user authentication, session management, and identity verification for the SoloCompass platform.
- **Owner:** Core Engineering
- **Status:** **Operational**

## 2. Product Role
- **Why this exists:** To ensure that only authorized users can access personal travel data, trips, and safety features.
- **What product problem it solves:** Protects user privacy, enables personalized experiences (Atlas AI, Buddies), and maintains the integrity of the safety check-in system.
- **What it should never do:** 
    - Expose sensitive user data (passwords, tokens) in logs or client-side code.
    - Allow brute-force access to accounts.
    - Reveal account existence through error messages (protected in forgot-password/resend-verification flows).

## 3. User Outcomes
- **Primary user value:** Secure, reliable access to travel itineraries, safety tools, and community features across multiple devices.
- **Secondary user value:** Recovery of account access if credentials are lost; persistent login across sessions via refresh tokens.
- **Trust / Safety / Business considerations:** Essential for the "Safety" brand promise; requires robust protection of user identity to maintain traveler confidence.

## 4. Scope
- **Included in scope:** User Registration (Email), Login (Password/Social), JWT Session Management (Access/Refresh), Password Reset, Failed Attempt Locking.
- **Out of scope:** Identity verification (Passport/ID scanning), Two-factor authentication (Planned for Future).

## 5. Functional Rules
- **Trigger conditions:** Visiting a protected route or submitting auth forms.
- **Inputs:** `email`, `password`, `name`.
- **Outputs:** `token` (Access), `refreshToken` (httpOnly Cookie), `user` profile object.
- **Priority rules:** Account verification is required for all non-admin users before login is permitted.
- **Dependencies:** Supabase/PostgreSQL, Resend API (Email), Infisical (Secrets).

## 6. UX Touchpoints
- **Routes:** `/login`, `/register`, `/forgot-password`, `/reset-password`, `/verify`.
- **Components:** `Login.jsx`, `Register.jsx`, `ProtectedRoute.jsx`, `authStore.js`.
- **Patterns:** 
    - **Loading:** Spinner shown during initialization (`authStore.initialize()`).
    - **Error:** Red alert banners in forms; 15-minute lockout for 5 failed attempts.

## 7. Technical Implementation
- **Frontend:** Zustand state management (`authStore.js`) with Axios interceptors for automatic token refresh.
- **Backend:** `backend/src/routes/auth.js`, `backend/src/middleware/auth.js`.
- **API Contract:** 
    - `POST /api/auth/login` -> `200 { user, token }`.
    - `POST /api/auth/register` -> `201 { success: true }`.
- **Data Model:** 
    - `users` table: Hashed passwords (BCrypt 12), roles, verification status.
    - `sessions` table: Hashed refresh tokens and device metadata.

## 8. Security & RBAC
- **Encryption:** Passwords salted and hashed with **BCrypt**.
- **Secrets:** All JWT keys managed via **Infisical (Vault)** for rotation safety.
- **Session Security:** `httpOnly`, `Secure`, `SameSite=Strict` cookies for refresh tokens to prevent XSS-stealing.
- **RBAC:** `requireAuth` middleware validates JWT signature and session existence in DB.

## 9. Performance & Caching
- **Token Strategy:** Short-lived Access Tokens (15m) + Long-lived Refresh Tokens (7d).
- **Latency Goal:** Auth verification and token refresh completes in < 200ms.
- **Fail-Soft:** In case of secondary DB lag, the JWT signature is still verified to allow read-only access (Fail-safe mode).

## 10. Observability & Monitoring
- **Metrics:** `FAILED_LOGIN_RATE` (monitored for lockout), `SESSION_ROTATION_FAIL`.
- **Logging:** All failed attempts and registration events logged with IP context (anonymized after 30 days).

## 11. Edge Cases & Failure Handling
- **Failure Modes:** Database downtime prevents authentication.
- **Recovery:** Automatic session refresh attempt on 401 responses; manual password reset via email.
- **Fail-Soft:** If Email service (Resend) is down, registration is queued and user is notified of delay.

## 12. Acceptance Criteria
- [x] Passwords are never stored in plain text.
- [x] Refresh token rotation is active to prevent session hijacking.
- [x] Rate limiting prevents brute force attacks on the login endpoint.

## 13. Changelog
- **v1.1:** Google Social Auth and Refresh Token rotation added.
- **April 8, 2026:** Documentation standardized for Final Gap Audit.

# SC-A02 API — AI Documentation

## 1. Overview
- **Type:** System / Feature
- **Primary purpose:** Provides intelligent, contextual travel guidance, automated itinerary generation, and real-time safety advice for solo travelers.
- **Owner:** Core Engineering / AI Team
- **Status:** **Operational**

## 2. Product Role
- **Why this exists:** Solo travelers often face "decision paralysis" or anxiety when planning alone. Atlas (SoloCompass AI) provides a 24/7 expert companion.
- **What product problem it solves:** Reduces time spent on manual research, offers instant safety clarifications, and provides highly personalized recommendations based on travel style.
- **What it should never do:** 
    - Provide definitive medical or legal advice.
    - Contradict official government travel advisories (FCDO).
    - Store personally identifiable information (PII) beyond what is necessary for the conversation context.

## 3. User Outcomes
- **Primary user value:** Instant access to tailored travel itineraries and destination-specific safety tips.
- **Secondary user value:** On-demand social and cultural advice (e.g., "How to meet people in Lisbon").
- **Trust / Safety / Business considerations:** Grounded in a "Safety-First" prompt architecture. Includes fallbacks to ensure users receive reputable advice even if the AI model is unavailable.

## 4. Scope
- **Included in scope:** Atlas AI Chat, Itinerary Generation (JSON), Safety Summaries, Quick Prompts Suggestion System, Result Caching.
- **Out of scope:** Direct flight/hotel bookings, Visual image generation.

## 5. Functional Rules
- **Trigger conditions:** Opening `AIChat.jsx` or requesting "Magic Itinerary".
- **Inputs:** Natural language queries, Contextual parameters (`destination`, `days`, `budget`, `pace`).
- **Outputs:** Markdown-formatted text, Structured JSON for itinerary parsing.
- **Priority rules:** Safety-related queries override creative suggestions with high-authority safety disclaimers.
- **Dependencies:** Azure OpenAI (GPT-4o), Internal cache table, `authStore`.

## 6. UX Touchpoints
- **Routes:** `/dashboard`, `/destinations`, `/trips/new`, `/safety`.
- **Components:** `AIChat.jsx`, `ItineraryGenerator.jsx`.
- **Patterns:** 
    - **Loading:** Typing indicator shown during API streaming.
    - **Fallback:** Rule-based responses triggered if API is unreachable.

## 7. Technical Implementation
- **Frontend:** React with Framer Motion; DOMPurify for secure markdown rendering.
- **Backend:** `backend/src/routes/ai.js`, `backend/src/services/aiService.js`.
- **API Contract:** 
    - `POST /api/ai/chat` -> `{ message, conversationId, destinationId? }`.
- **Data Model:** `ai_itinerary_cache` table (destination, days, pace, itinerary_json).
- **External Integrations:** Azure OpenAI API via LiteLLM bridge.

## 8. Security & RBAC
- **Plan Enforcement:** Navigator (Unlimited), Guardian (50/day), Explorer (1/trip).
- **Rate Limiting:** `aiQueryLimiter` restricts per-IP usage to prevent resource exhaustion.
- **Content Security:** All output is sanitized via **DOMPurify** to prevent persistent XSS from malicious model injections.

## 9. Performance & Caching
- **Itinerary Cache:** Stored in DB with a **24-hour TTL**.
- **Latency Management:** Streaming responses used for chat to reduce TTFB.
- **Fail-Soft:** `getFallbackResponse()` in `aiService.js` provides static, high-quality safety data if the model is offline.

## 10. Observability & Monitoring
- **Metrics:** `AI_LATENCY`, `AI_TOKEN_USAGE` (per user), `AI_FALLBACK_COUNT`.
- **Logging:** All chat requests logged to server-side telemetry with user context.

## 11. Edge Cases & Failure Handling
- **Failure Modes:** API Throttling, Model Hallucination, Connectivity loss.
- **Recovery:** In-chat retry mechanism; fallback to rule-based safety instructions.
- **Audit:** Automated flagging of low-confidence safety responses for admin review.

## 12. Acceptance Criteria
- [x] Itineraries are valid JSON.
- [x] Safety queries always include a standard disclaimer.
- [x] Responses are streamed for improved UX.

## 13. Changelog

# SC-A03 API — Trips Documentation

## 1. Overview
- **Type:** System / Feature
- **Primary purpose:** Core management of solo travel itineraries, tracking trip execution, and ensuring travel readiness.
- **Owner:** Core Engineering
- **Status:** **Operational**

## 2. Product Role
- **Why this exists:** Travel planning is complex and fragmented. This system centralizes logistics, AI-generated plans, and safety preparedness in one "Mission File."
- **What product problem it solves:** Eliminates the need for multiple apps for itineraries, checklists, and documentation. Provides active trip tracking and readiness scores.
- **What it should never do:** 
    - Lose itinerary version history during regeneration.
    - Delete trip data without user confirmation.
    - Expose private itineraries to other users.

## 3. User Outcomes
- **Primary user value:** A single, structured source of truth for all solo adventures, from planning to completion.
- **Secondary user value:** "Safety Readiness" scores that give peace of mind before departure; version control to experiment with AI plans.
- **Trust / Safety / Business considerations:** Direct link between trip dates and safety monitoring (automated check-in activation).

## 4. Scope
- **Included in scope:** Trip CRUD, Itinerary Management (Daily breakdown), Version Control (Rollback), Readiness Monitoring (Checklists), Status Automation (Planning -> Active -> Completed), PDF Export.
- **Out of scope:** Direct social sharing (planned), Real-time GPS tracking (handled by A11 Check-ins).

## 5. Functional Rules
- **Trigger conditions:** New trip creation, Date-based status transitions, AI Generation requests.
- **Inputs:** `name`, `destination`, `start_date`, `end_date`, `budget`, `activities`.
- **Outputs:** Structured itinerary tree, Readiness Score (0-100%), PDF summaries.
- **Priority rules:** Confirmed/Active trips are prioritized in the UI over Planning/Cancelled trips.
- **Dependencies:** Itinerary AI Service, Readiness Service, Budgets Table.

## 6. UX Touchpoints
- **Routes:** `/trips`, `/trips/:id`, `/trips/new`.
- **Components:** `TripCard.jsx`, `ItineraryTimeline.jsx`, `ReadinessWidget.jsx`.
- **Patterns:** 
    - **Empty:** Encourages creating the first "Mission."
    - **Loading:** Skeleton cards for trip lists.
    - **Error:** API Error Boundary with "Try Again" fallback.

## 7. Technical Implementation
- **Frontend:** Trips.jsx, TripDetail.jsx. Uses Axios with cancel tokens for stable navigation between trip segments.
- **Backend:** `backend/src/routes/trips.js` with transaction-safe nested creation logic.
- **API Contract:** 
    - `POST /api/trips` -> `{ title, destination, startDate, endDate, budget? }`.
    - `GET /api/trips/:id` -> Returns full trip tree (Days + Activities).
- **Data Model:** 
    - `trips`: Base metadata with status enum.
    - `itinerary_days`: O(n) relationship to trips.
    - `activities`: O(n) relationship to days.
    - `itinerary_versions`: Snapshots of the full JSON tree for rollback capability.

## 8. Security & RBAC
- **Ownership:** Strict `userId` match required for all GET/PUT/DELETE operations; cross-user ID spoofing is blocked at the middleware level.
- **Plan Limits:** Explorer (Free) restricted to 2 active trips; Navigator (Premium) is unlimited.
- **Data Integrity:** `ON DELETE CASCADE` enforced via DB constraints to prevent orphaned activities.

## 9. Performance & Caching
- **Eager Loading:** Uses SQL joins to fetch trip + days + activities in a single query to eliminate N+1 latency issues.
- **Caching Strategy:** Trip metadata is cached for 10 minutes; Itinerary details are cached for 30 minutes unless `forceRefresh` is requested.
- **Fail-Soft:** If the Itinerary AI service is down, the system allows manual editing of previous versions to ensure the user is never blocked.

## 10. Observability & Monitoring
- **Metrics:** `TRIP_CREATE_LATENCY`, `READINESS_SCORE_AVG`, `ITINERARY_REGEN_RATE`.
- **Audit:** All status transitions (Planning -> Active) are logged for lifecycle analysis.

## 11. Edge Cases & Failure Handling
- **Failure Modes:** Itinerary generation timeout, PDF generation memory overflow.
- **Recovery:** Itinerary versioning allows recovery from destructive edits; Background job retries for AI generation.
- **Fail-Safe:** System defaults to the last locally cached version of the itinerary if the network is lost mid-trip.

# SC-A04 API — Accommodations Documentation

## 1. Overview
- **Type:** System / Feature
- **Primary purpose:** Centralized management of accommodation details (Base Camp) for solo trips.
- **Owner:** Core Engineering
- **Status:** **Operational**

## 2. Product Role
- **Why this exists:** Solo travelers need a single "safe haven" for their addresses and confirmation codes, especially when arriving in new cities.
- **What product problem it solves:** Replaces the need to search through emails for check-in times and physical addresses.
- **What it should never do:** 
    - Handle financial transactions or bookings (tracking system only).
    - Expose physical addresses to unauthorized users.

## 3. User Outcomes
- **Primary user value:** Instant, offline-ready access to lodging info (Base Camp).
- **Secondary user value:** Automatic contribution to the trip budget and "Readiness" checklist completion.
- **Trust / Safety / Business considerations:** Essential for safety; lodging info is a key part of the emergency data shared with trusted contacts.

## 4. Scope
- **Included in scope:** Logistics Tracking (Name, Type, Address), Confirmation Numbers, Check-in/out Validation, Budget/Currency sync.
- **Out of scope:** Multi-stop lodging (currently 1 per trip), Direct Bookings/Reserves.

## 5. Functional Rules
- **Trigger conditions:** User clicks "Add Accommodation" in Trip Details.
- **Inputs:** `name`, `type`, `check_in_date`, `check_out_date`, `cost`, `currency`.
- **Outputs:** Sanitized JSON lodging record.
- **Priority rules:** If a conflict arises (Accommodation already exists), the system prompts for an update rather than creating a duplicate.
- **Dependencies:** Trips Table (Ownership check).

## 6. UX Touchpoints
- **Routes:** `/trips/:id` (Trip Details).
- **Components:** `TripDetail.jsx`, `AccommodationCard`.
- **Patterns:** 
    - **Empty:** "No Base Camp set" prompt.
    - **Error:** API toast notifications for validation errors (e.g., dates outside trip range).

## 7. Technical Implementation
- **Frontend:** `TripDetail.jsx` manages the local state `accommodations`.
- **Backend:** `backend/src/routes/accommodations.js`.
- **API Contract:** 
    - `POST /api/trips/:id/accommodation` -> `{ name, type, address, checkIn, checkOut, cost, currency }`.
- **Data Model:** 
    - `accommodations`: `trip_id`, `name`, `type`, `address`, `check_in_date`, `check_out_date`, `confirmation_number`, `cost`, `currency`.
- **Constraints:** `trip_id` has a **UNIQUE** index ensuring one primary accommodation per trip.

## 8. Security & RBAC
- **Data Isolation:** Accommodation data is isolated by `userId` (joined via `trips` table) to prevent lateral access.
- **Sanitization:** String inputs are sanitized for notes/address to prevent persistent XSS.
- **Permissions:** `requireAuth` + `isTripOwner` middleware enforced for all operations.

## 9. Performance & Caching
- **Caching:** Accommodation data is included in the Eager-Loaded Trip Detail fetch and cached for **10 minutes** at the client level.
- **Latency Goal:** Lodging updates complete in < 300ms.
- **Fail-Soft:** If the DB update fails, the UI retains the draft in local state to prevent data loss.

## 10. Observability & Monitoring
- **Metrics:** `ACCOMMODATION_ADD_RATE`, `DATE_VALIDATION_FAILURE_COUNT`.
- **Audit:** All lodging coordinate changes are logged to the safety engine for last-seen-location auditing.

## 11. Edge Cases & Failure Handling
- **Failure Modes:** 409 Conflict (Duplicate entry), Invalid date range (Before trip start).
- **Recovery:** Inline form validation highlights errors before submission.
- **Logging:** All destructive edits (DELETE) are soft-logged for 30 days.

# SoloCompass Technical Documentation — Audit Strengthening Report (Full Library)

This document contains the authoritative technical specifications, architectural mappings, and operational triggers established during the April 2026 Audit. Use this as the "Source of Truth" for engineering implementation and QA automation.

---

## 🏗️ GLOBAL API STANDARDS

### Standard Response Schema
All SC-API responses adhere to the following JSON envelope:
```json
{
  "success": boolean,
  "data": object | null,
  "error": {
    "code": string,      // e.g., "RATE_LIMIT_EXCEEDED"
    "message": string,  // Human readable
    "details": object   // Optional validation errors
  } | null,
  "status": number       // HTTP Status Code
}
```

### Common Status Codes
- `200/201`: Success / Resource Created.
- `403`: Permission Denied (RBAC / Paywall).
- `422`: Validation Error (Invalid schema).
- `429`: Rate Limit Exceeded.
- `503`: External Provider Timeout (Fail-soft triggered).

---

## 🛡️ API SECURITY & INFRASTRUCTURE ADDITIONS (A01 - A36)

### SC-A01 — Auth
- **Service Interaction:** UI -> Auth Route -> Bcrypt Helper -> Infisical (JWT Secret) -> SQLite/Postgres -> Return.
- **RBAC:**
  - **Explorer/Guest:** Can read public profiles.
  - **All Auth:** Can manage own session.
- **Contract:** `POST /login`: `{ email (req), password (req) }`.
- **Performance:** JWT Access Token (15m TTL).
- **Monitoring Trigger:** Alert Dev Team if `FAILED_LOGIN_RATE` > 20/min per IP (Potential Brute Force).

### SC-A02 — AI (Atlas)
- **Service Interaction:** UI -> AI Route -> RBAC Middleware (Tier Check) -> LiteLLM Proxy -> Azure OpenAI -> DOMPurify -> Return.
- **RBAC:**
  - **Explorer:** 1 free query per trip.
  - **Navigator/Guardian:** Unlimited safety queries.
- **Performance:** Itinerary cache (24h TTL); Safety Index summaries (24h TTL).
- **Monitoring Trigger:** Log `AI_FALLBACK_COUNT`. Alert if > 10% of queries use fallback mock due to timeout.

### SC-A03 — Trips
- **Infrastructure:** Transactional integrity for trip initialization.
- **Contract:** `POST /api/trips`: `{ title, destination, startDate, endDate, budget? }`.
- **Security:** Strict `userId` ownership check on all CRUD operations.
- **Observability:** `TRIP_CREATE_LATENCY` monitoring.

### SC-A04 — Accommodations
- **Contract:** `POST /api/trips/:id/accommodation`.
- **Constraints:** `trip_id` UNIQUE index ensures 1 accommodation per trip.

### SC-A05 — Bookings
- **Contract:** `POST /api/trips/:id/bookings`.
- **Data Isolation:** SQL `EXISTS` checks for cross-table security.

### SC-A06 — Trip Documents
- **Service Interaction:** UI -> Documents Route -> Ownership Check -> Supabase Storage -> DB (URL Save) -> Return.
- **RBAC:**
  - **Explorer:** Max 5MB per file; 50MB total.
  - **Navigator:** Max 20MB per file; 500MB total.
- **Constraints:** `trip_id` (FK) + `filename` (UNIQUE per trip).
- **Monitoring Trigger:** Alert if `SUPABASE_UPLOAD_FAILURE_RATE` > 2% in 15 mins.

### SC-A07 — Trip Places
- **Infrastructure:** Google Places API sync.
- **Constraints:** `trip_id` + `place_id` UNIQUE constraint prevents duplicate pins.

### SC-A08 — Destinations
- **Infrastructure:** LiteLLM / Azure AI.
- **RBAC:** `premiumOnly` check for AI agent interaction.
- **Caching:** AI Intelligence briefings cached for 24h.

### SC-A09 — Users
- **Infrastructure:** GDPR-compliant `GET /export` (Flattened JSON archive).
- **Constraints:** `users.email` UNIQUE Case-Insensitive.

### SC-A10 — Advisories
- **Infrastructure:** **FCDO_RSS_SYNC** task (4-hour interval).
- **Caching:** Government alerts cached with 1-hour TTL.
- **Observability:** `SYNC_HEARTBEAT` monitoring.

### SC-A11 — Check-In (Safety Hub)
- **Service Interaction:** CRON Sweeper -> DB Search (Missed Time) -> Escalation Service -> Twilio (SMS) / Resend (Email) -> DB Update (Status: Escalated).
- **Orchestration:** **Node-Cron Service** (Stateful). On restart, "Sweeper" scans for any `scheduled_time < now` and `status == 'pending'`.
- **Monitoring Trigger:** **CRITICAL ALERT** if SMS failure rate > 5% in 1 hour. Trigger P1 Incident.
- **Error Handling:** If Twilio returns 60200 (Invalid Phone), mark contact as `INACTIVE_VERIFICATION_REQUIRED` and notify traveler via Push.

### SC-A12 — Safety
- **Performance:** Pre-rendered crime heatmaps at city-grid level.
- **Fallback:** Automatic transition to regional stats if high-precision data fails.

### SC-A13 — Emergency Contacts
- **Infrastructure:** **Twilio SMS Gateway** integration.
- **Security:** Phone numbers masked in logs; 8-char random verification codes.

### SC-A14 — Billing
- **Infrastructure:** **Stripe Webhook Listener** with `Stripe-Signature` validation.
- **RBAC:** Plan-gating synced to `users.subscription_tier` enum.

### SC-A15 — Admin
- **RBAC:** `requireAdmin` middleware checks `user.role === 'admin'`.
- **Audit:** `audit_logs` table records all admin actions and reasons.

### SC-A16 — Analytics
- **Contract:** `POST /api/telemetry` -> `{ event, properties }`.
- **Infrastructure:** Events in-memory queuing with 5s flush.
- **Observability:** `MAU` and `Retention` tracking.

### SC-A17 — Notifications
- **Infrastructure:** **Socket.io** for real-time; REST for history.

### SC-A19 — Currency
- **Infrastructure:** Redundant fallback to `api.exchangeratesapi.io`.
- **Caching:** Rates stored in `rates_cache` for 12 hours.

### SC-A20 — Weather
- **Performance:** Data grouped by `city_id` to prevent redundant API calls.
- **Caching:** Current (30m), Forecast (3h).

### SC-A26 — Exchange
- **Contract:** `GET /api/exchange/convert?from=USD&to=EUR`.
- **Infrastructure:** Redundant rates provider strategy.

### SC-A32 — Flights
- **Service Interaction:** UI -> Flight Route -> AviationStack API -> Local Cache -> Return.
- **Performance:** **5-minute TTL** for live status (Gate/Terminal).
- **Fail-Soft:** If AviationStack is 5xx, return last cached status with `is_stale: true` flag in the `data` envelope.

### SC-A34 — Webhooks
- **Security:** HMAC-style signatures (**Stripe-Signature** pattern).
- **Verification:** IP-Whitelisting for incoming platform IPs.

### SC-A35 — SMS Webhook
- **Security:** Signature validation using **Twilio Request Validation** SDK.
- **Keywords:** `YES` (Verify), `STOP` (Unsubscribe), `START` (Re-verify).

### SC-A36 — Translate
- **API:** **Azure AI Translator**.
- **Security:** Rate limiting per user to prevent massive scraping abuse.

## 🧱 COMPONENT ARCHITECTURE & STATE ADDITIONS (C01 - C26)

- **Patterns:** 
    - **Active:** Icon and label change color to `primary` (#0ea5e9).
    - **Transition:** Slide-down 100% when hidden; Spring-bounce when revealed.

### 7. Technical Implementation
- **Frontend:** `frontend/src/components/BottomNav.jsx`.
- **State Management:** Uses a local `isVisible` state driven by a scroll listener.

# SC-DB-01 — Database & Schema Handbook

This document serves as the single source of truth for the SoloCompass data layer. It covers schemas, relationships, indexing strategies, and migration patterns.

## Entity Relationship Diagram (ERD)

```mermaid
erDiagram
    USERS ||--o{ TRIPS : "plans"
    USERS ||--|| PROFILES : "has"
    USERS ||--o{ NOTIFICATIONS : "receives"
    TRIPS ||--o{ ITINERARY_DAYS : "contains"
    ITINERARY_DAYS ||--o{ ACTIVITIES : "holds"
    TRIPS ||--o{ ACCOMMODATIONS : "includes"
    TRIPS ||--o{ BOOKINGS : "tracks"
    USERS ||--o{ CHECK_INS : "submits"
    USERS ||--o{ EMERGENCY_CONTACTS : "notifies"
    DESTINATIONS ||--o{ REVIEWS : "collects"
    USERS ||--o{ BUDGET_ENTRIES : "logs"
```

## Core Schema Definitions

### 1. User Identity (`users`)
| Column | Type | Constraints | Purpose |
| :--- | :--- | :--- | :--- |
| `id` | SERIAL | PK | Unique identifier |
| `email` | TEXT | UNIQUE, NOT NULL | Login credential |
| `password` | TEXT | NOT NULL | BCrypt hashed password |
| `is_premium` | BOOLEAN | DEFAULT false | Feature gating flag |
| `subscription_tier`| TEXT | DEFAULT 'free' | Plans: free, navigator, guardian |

### 2. Safety & Monitoring

#### `scheduled_check_ins`
The heart of the safety system. This table is polled by the **Sweeper Service**.
- `scheduled_time`: When the user is expected to check in.
- `escalation_level`: Current stage of escalation (0: None, 1: Push, 2: SOS).
- `is_active`: Toggle for the monitor.

#### `emergency_contacts`
- `relationship`: Personal connection (Family, Friend, etc.).
- `delivery_order`: `email_first` or `sms_first`.
- `verified`: Boolean for verified phone/email.

### 3. Trip Orchestration

#### `trips`
- `status`: `planning`, `confirmed`, `completed`, `cancelled`.
- `solo_vibe`: User-selected pace for AI generation.

#### `itinerary_days` & `activities`
Nested components for granular travel scheduling.

## Indexing & Performance Strategy

### Critical Indexes
1. `idx_users_email`: B-Tree index on `users(email)` for sub-millisecond logins.
2. `idx_checkins_scheduled`: Index on `scheduled_check_ins(scheduled_time)` where `is_active = true`. This is optimized for the 60s Sweeper cycle.
3. `idx_trips_user_id`: Foreign key index to speed up Dashboard loading.

## Migration Patterns
SoloCompass uses raw SQL migration scripts located in `backend/migrations/`. 
- **Rule**: All table modifications MUST be idempotent (`CREATE TABLE IF NOT EXISTS`).
- **Data Integrity**: Foreign keys use `ON DELETE CASCADE` for parent-child cleanup (e.g., deleting a user purges their trips).

## 🏗️ GLOBAL API STANDARDS

### Standard Response Schema
All SC-API responses adhere to the following JSON envelope:
```json
{
  "success": boolean,
  "data": object | null,
  "error": { "code": string, "message": string, "details": object } | null,
  "status": number
}
```

### Common Status Codes
- `200/201`: Success.
- `403`: Forbidden (Quota/Plan reached).
- `422`: Validation error.
- `503`: Fallback triggered (External API down).

- **Tooltips:** "Database: Primary", "Security Vault: Sealed".

### 8. Technical implementation
- **Frontend:** React functional component using `framer-motion` for pulse/ping animations.
- **Infrastructure:** Direct ping to `/health` endpoint every 60s.
- **Security:** Endpoint returns non-privileged payload: `{ status: 'ok', subsystems: { db: true, vault: true } }`.
- **Dependencies:** `authStore`, `api` wrapper.

### 9. Security
- **Middleware:** None (client-side display), but `/health` endpoint is usually public or lightweight.

### 10. Edge cases and failure handling

- **Degraded DB:** API returns `status: 'degraded'` if DB connection is intermittent.

### 11. Acceptance criteria

# SC-OPS-01 — Infrastructure & DevOps Guide

This document outlines the orchestration of the SoloCompass deployment environment, including CI/CD pipelines, containerization, and secret management.

## Deployment Stack

| Component | Technology | Provider |
| :--- | :--- | :--- |
| **Frontend** | React / Vite | Vercel |
| **Backend** | Node.js / Express | Render (Docker) |
| **Database** | PostgreSQL | Supabase |
| **Secrets** | Infisical SDK | Infisical |
| **Storage** | S3 / Supabase | Supabase Bucket |

## Containerization (Backend)

The backend is packaged using a standard multi-stage Dockerfile:
- **Base**: `node:18-alpine`
- **Port**: `3005`
- **Runtime**: Runs `npm start` which initiates the database initialization and the Sweeper monitoring service.

## CI/CD Pipeline (GitHub Actions)

Located in `.github/workflows/`:

1. **`main.yml`**: Triggers on push to `main`. Coordinates the overall deployment pulse.
2. **`backend.yml`**: 
    - Runs Jest unit tests.
    - Builds the Docker image.
    - Pushes to Render via webhook trigger.
3. **`frontend.yml`**:
    - Runs linting and Vite build checks.
    - Deploys preview branches to Vercel.
4. **`migration.yml`**: 
    - Automated schema updates against the Supabase production instance.

## Secret Hydration (Infisical)

SoloCompass uses **Zero-Trust Secret Management**. No `.env` files are stored in production.
- **Boot Sequence**: `backend/src/services/infisical.js` (hypothetically) fetches secrets at runtime using `INFISICAL_CLIENT_ID` and `CLIENT_SECRET`.
- **Injected Keys**: 
    - `DATABASE_URL`
    - `STRIPE_SECRET_KEY`
    - `RESEND_API_KEY`
    - `AZURE_OPENAI_API_KEY`

## Logging & Observability
- **Log Source**: `backend/logs/` (Local development) / Standard Output (Production).
- **Format**: JSON structured logging for ingestion into CloudWatch/Logtail.
- **Alerting**: Webhook triggers on `P0` errors (Database down, Safety Sweeper stall).

# SC-SEC-01 — Security & Compliance Manifesto

SoloCompass is a travel safety application. Our security posture is critical to user trust and physical safety. This document details our defense-in-depth strategy.

## 1. Authentication & Authorization (RBAC)

### Password Policy
- **Hashing**: `bcrypt` with a salt factor of 12.
- **Complexity**: Enforced at the frontend and re-validated at the backend.
- **Lockout**: 5 failed attempts trigger a 30-minute IP-based cooldown.

### Authorization
- **Mechanism**: JWT (JSON Web Tokens) with a 15-minute expiration.
- **Refresh**: Stateful 7-day refresh tokens stored in `HttpOnly, Secure, SameSite=Lax` cookies to prevent XSS.
- **RBAC Matrix**: 
    - `Explorer`: Read, basic trip create.
    - `Navigator`: AI features, premium data access.
    - `Admin`: User management, global SOS monitoring.

## 2. Data Protection

### Transport & Storage
- **TLS**: 100% of traffic is encrypted via TLS 1.3 (Vercel/Render).
- **Data at Rest**: Supabase PostgreSQL uses AES-256 transparent disk encryption.
- **PII Handling**: User phone numbers and emergency contacts are treated as High-Sensitivity data.

### Compliance
- **GDPR**: Users have the right to "Export Trip Data" and "Permanent Account Deletion" (SVC: `users.js` handles cascading deletes).
- **PCI-DSS**: SoloCompass **never** touches raw credit card data. All payments are handled via Stripe Elements (PCI-DSS Level 1 Service Provider).

## 3. Threat Mitigation

### SQL Injection
- **Strategy**: Prepared statements and parameterized queries via the `pg` driver.
- **Helper**: `backend/src/db.js` `convertPlaceholders` ensures no raw string concatenation is possible.

### XSS & Injection
- **Frontend**: React-based auto-escaping of JSX.
- **AI Content**: All Atlas AI responses are sanitized using `DOMPurify` before being rendered in the `AIChat.jsx` component.

### Rate Limiting
- **Global**: 100 requests per 15 minutes per IP.
- **Sensitive (Reset Password)**: 3 requests per hour to prevent brute-force recovery.

## 4. Incident Response
- **Logging**: All `401 Unauthorized` and `403 Forbidden` events are logged with IP metadata.
- **SOS Validation**: SOS Slider (C05) requires a 3-second hold to prevent accidental triggers while ensuring zero-click difficulty in distress situations.

---

- [x] JWT lifecycle mapped
- [x] PII and PCI-DSS compliance defined
- [x] SQLi and XSS defenses documented

#### 11. Edge Cases & Fail-Soft
- **Fail-Soft:** If Database is unreachable, the system allows read-only access for users with valid, non-expired JWTs.

# SC-A34 API — Webhooks Documentation

## 1. Overview
- **Type:** External System Bridge
- **Primary purpose:** Processing asynchronous events from external service providers (Stripe, Twilio, Firebase).
- **Owner:** DevOps / Infrastructure
- **Status:** **Operational**

## 2. Product Role
- **Why this exists:** To ensure the system stays in sync with external actions (like payments or SMS replies) without polling.
- **What product problem it solves:** Long-wait times for subscription updates; ensures "Real-time" state across distributed services.

## 3. User Outcomes
- **Primary user value:** Instant activation of premium features after purchase.
- **Secondary user value:** Real-time confirmation of emergency contact verification.

## 4. Scope
- **Included in scope:** Stripe (Payments/Subs), Twilio (SMS Status), SendGrid/Resend (Email Bounce).
- **Out of scope:** Outbound webhook notifications (planned for B2B partners).

## 5. Functional Rules
- **Trigger:** Incoming HTTP POST from authorized provider.
- **Processing Logic:** 
    1. Validate Signature.
    2. Route to specific handler (e.g., `handleStripeEvent`).
    3. Update internal DB state.
    4. Trigger Notification (SC-A17) if required.

## 6. UX Touchpoints
- **Indirect:** Users see the result (e.g., "Subscription Active!") via subsequent UI refreshes or push alerts.

## 7. Technical Implementation
- **Endpoints:** `/api/webhooks/stripe`, `/api/webhooks/sms`.
- **Infrastructure:** Signature verification using provider SDKs (e.g., `stripe.webhooks.constructEvent`).
- **Idempotency:** Unique `event_id` tracking in `webhook_logs` to prevent double-processing.

## 8. Security & RBAC
- **Hardening:** **Raw Body Buffering** required for signature verification; IP Whitelisting for platform IPs.
- **RBAC:** Internal system-only access (public endpoints protected by cryptographic signatures).

## 9. Performance & Caching
- **Concurrency:** Uses a 200 OK "Fast Response" pattern—acknowledge receipt first, then process asynchronously via queuing.

## 10. Observability & Monitoring
- **Metrics:** `WEBHOOK_VERIFICATION_FAILURE_RATE`, `EVENT_PROCESSING_LATENCY`.
- **Alert:** Critical alert for P1 if Stripe signature validation fails consistently (Potential attack).

## 11. Edge Cases & Failure Handling
- **Failure Modes:** Signature mismatch; Invalid payload.
- **Recovery:** Logging of raw payloads for 7 days allows for manual re-processing of failed events.

## 12. Acceptance Criteria
- [x] Test webhooks successfully update the `users.subscription_tier`.
- [x] Replayed events are ignored via idempotency check.

# SC-SEC-01 — Security & Compliance Manifesto
SoloCompass treats user location and emergency data as P0 High-Sensitivity assets.

### 1. Authentication & Authorization (RBAC)
- **Hashing**: `bcrypt` with a salt factor of 12.
- **Lockout**: 5 failed attempts trigger a 30-minute IP-based cooldown.
- **Mechanism**: JWT (JSON Web Tokens) with 15-minute Access / 7-day Refresh tokens.
- **Refresh Strategy**: Stored in `HttpOnly, Secure, SameSite=Lax` cookies to prevent XSS.

### 2. Data Protection
- **TLS**: 100% of traffic is encrypted via TLS 1.3.
- **Compliance**: GDPR-compliant cascade deletes (`users.js`). PCI-DSS 100% delegated to Stripe Elements.
- **XSS & Injection**: All Atlas AI responses and UGC are sanitized via **DOMPurify** before rendering.

# SC-DB-01 — Database & Persistence Handbook
- **Primary Persistence**: PostgreSQL 15 (Supabase).
- **Critical Indices**:
    - `idx_users_email`: Authentication speed.
    - `idx_checkins_scheduled`: Critical for the 60s CRON Sweeper loop.
    - `idx_trips_user_id`: Dashboard responsiveness.
